🐜 Sqrt Decomposition · Offline Query Batching

Mo's Algorithm

Sort the queries. Crawl the array once. Never recompute from scratch.

Given all your range queries upfront, reorder them by √n-sized blocks so a single two-pointer crawl — extending and shrinking the window by one element at a time — answers every single one. Total pointer movement stays bounded at O((n+Q)√n), often far cheaper than a segment tree per query.

🐜 Why the Block-Sort Order Matters

🧱

Sqrt-Sized Blocks

Divide the array's indices into blocks of size ≈√n. Sort queries first by which block their left endpoint falls in, then by right endpoint. This groups queries with nearby left bounds together, bounding how far the left pointer ever has to travel.

🐌

Incremental, Never From Scratch

The current window's answer updates by adding or removing exactly ONE element as a pointer moves one step — never recomputed from zero. This is what makes the total cost proportional to total pointer distance traveled, not to query count times array size.

📐

The O((n+Q)√n) Proof

Within a block, right-pointer movement is bounded by O(n) total across all queries in that block (queries sorted by r). Left-pointer movement is bounded by O(√n) per query since it can't leave its own block's neighborhood. Sum both bounds across all √n blocks and all Q queries.

🏆

Where This Wins

Mo's Algorithm shines for range queries that are easy to update incrementally (distinct count, mode, sum of squares) but hard to maintain in a segment tree. It requires knowing all queries upfront (offline) — the fundamental tradeoff versus online structures.
